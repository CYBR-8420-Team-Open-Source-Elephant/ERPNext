3. Collision Attacks  [State: Mitigation Implemented]  [Priority: High]

24. Authorization Bypass  [State: Not Applicable]  [Priority: High] 

Category:Information Disclosure
Description:Can you access ERPNext Records and bypass the permissions for the object? For example by editing the files directly with a hex editor, or reaching it via filesharing? Ensure that your program is the only one that can access the data, and that all other subjects have to use your interface.
Justification:This threat should be dealt with in the implementation of MariaDB and is out of scope for ERPNext. The only mitigation that can be done by ERPNext is to keep the database management system up to date with the latest security patches.

25. Data Flow Sniffing  [State: Mitigation Implemented]  [Priority: High] 

Category:Information Disclosure
Description:Data flowing across Business Transaction Record may be sniffed by an attacker. Depending on what type of data an attacker can read, it may be used to attack other parts of the system or simply be a disclosure of information leading to compliance violations. Consider encrypting the data flow.
Justification:Mitigation of this threat can depend on the individual deployment of ERPNext. If the Backend Web Server and database instance are on the same trusted host or isolated network, such as in the containerized setup, encryption is unnecessary (see the default Docker Compose file: https://github.com/frappe/frappe_docker/blob/main/pwd.yml). MariaDB can also be configured to encrypt in-transit data using TLS (https://mariadb.com/docs/server/security/securing-mariadb/encryption/data-in-transit-encryption/secure-connections-overview).

26. Potential Excessive Resource Consumption for 2.0 ERPNext Backend Web Server or ERPNext Records  [State: Needs Investigation]  [Priority: High] 

Category:Denial Of Service
Description:Does 2.0 ERPNext Backend Web Server or ERPNext Records take explicit steps to control resource consumption? Resource consumption attacks can be hard to deal with, and there are times that it makes sense to let the OS do the job. Be careful that your resource requests don't deadlock, and that they do timeout.
Justification:The Frappe framework supports rate limiting, which can help mitigate denial of service attacks on the backend web server (https://docs.frappe.io/framework/user/en/rate-limiting). It also allows for the configuration of HTTP timeouts (https://docs.frappe.io/framework/user/en/bench/bench-commands#config-commands). Further investigation is needed to determine if the application limits total resource consumption in any way, or if this is expected to be handled by the operating environment.
